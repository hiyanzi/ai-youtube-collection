03 / 技術重點

Gemini 直接吃 YouTube URL

以往要用 AI 摘要影片，需要先寫字幕爬蟲、處理無字幕的 OCR fallback、甚至接 Whisper 做語音識別⋯⋯

❌ 傳統做法

抓字幕 → 清洗 → 斷句 → 送 LLM → 組合結果

開發成本高、工程鏈脆弱、無字幕影片直接失敗。

✅ 本專案做法

Gemini 2.5 Flash 直接接受 YouTube URL 作為多模態輸入，在 Google 後端解讀視聽內容。

150 行腳本搞定，每支成本不到 $0.03。

03 / 技術重點

嚴格的 Prompt 工程

LLM 天生會「討好使用者」— 不加限制，會把所有影片的 production_signals 都判成 true，分數也一律 8+ 分。

💰 生產信號規則

影片必須明確花 30 秒以上專門討論該面向（成本、延遲、可靠性⋯），才能標記為 true。

光提一句或列在投影片 — 一律 false。

📊 分數天花板

依 content_type 設定上限：
· 理論講解 → 最高 4 分
· 新聞 → 最高 2 分
· 討論雜談 → 最高 3 分

杜絕 LLM 對純概念影片的灌水。

03 / 技術重點

斷點續傳與容錯

💾

每支立即存檔

每完成一支影片的 AI 摘要，立刻寫入 JSON。中途網路斷、Rate Limit、Ctrl+C 都不會掉資料。再跑時自動跳過已摘要的影片。

🛡

超長影片 Fallback

當影片內容超過 Gemini Token 上限，自動降級寫入 Fallback 標記並跳過，避免自動更新管線卡死。

「對長時間任務，失敗能重來 比 不會失敗 更重要。」

04 / 學習

核心學習

  1. LLM 的「客觀性」需要被工程化。
    不加限制，LLM 會給出討好式的高分。只有嚴格的負向規則與分數天花板，才能產出有決策參考價值的數據。
  2. Schema 先行，UI 後設計。
    花一個下午把 JSON Schema 想清楚，前端的 React state 與篩選邏輯就變得極其輕量，沒有任何 hardcoded 的欄位。
  3. 選對工具能讓複雜度消失。
    Gemini 能直接吃 URL 這件事，讓整個字幕爬蟲 + OCR + Whisper 的三天工程直接變成 150 行腳本。
